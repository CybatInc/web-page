User Guide_

The Intelligence Dashboard is your real-time command center for autonomous threat monitoring. This guide covers every panel, how the engine processes threats, and how enforcement actions are applied.

Open Dashboard →

1. Dashboard Layout

After logging in at intelligence.cybat.co, you land on the main Intelligence Dashboard, which consists of three areas:

3D Threat Globe

A rotating Earth visualization showing the geographic origin of all detected threats. Glowing lavender points represent high-severity sources; mint points represent medium-severity. Click any point to see the full list of incidents from that location.

[ screenshot-globe.png ]

Incident Feed

A live list of all detected threats sorted by recency. Each card shows the source IP, threat level (high / medium), timestamp, affected GCP project, and the enforcement action taken. Click any incident to open the full detail panel.

[ screenshot-incidents.png ]

Incident Detail Panel

Opens over the feed. Shows the full Gemini AI analysis narrative, all log evidence records that triggered the detection, the enforcement decision and its status, and the Slack notification dispatched to your team.

[ screenshot-detail.png ]

2. How the Engine Processes a Threat

Each log record moves through the following pipeline automatically. The entire sequence completes in under 5 seconds from log emission to enforcement.

01
Ingestion — Cloud Logging routes your project's logs to a dedicated Pub/Sub topic. The Cybat ingestion layer (Fluent Bit + pubsub-bridge running on GKE) pulls messages continuously and forwards them to the SOC agent.
02
IP Grouping — Logs from the same batch are grouped by source IP to build a per-IP request picture before analysis.
03
Subscription Check — The engine validates your organization's active Marketplace subscription via GCP Service Control before proceeding.
04
Verdict Cache — IPs seen in the last 24 hours skip Gemini re-analysis and are enforced immediately using the cached verdict, eliminating redundant AI calls.
05
Gemini Analysis — New IPs are analyzed by Vertex AI Gemini 2.5 Flash Lite. The model receives the full log context and returns a structured verdict: threat level (high / medium / low / none) and a human-readable explanation.
06
Enforcement — For high and medium verdicts, the engine attempts VPC Firewall deny first (Layer 3/4). If the VPC network is not discoverable from the logs, it falls back to Cloud Armor IP block (Layer 7). Enforcement is targeted to the specific GCP project the logs originated from.
07
Persistence & Alerting — The incident, AI analysis, and evidence records are written to the dashboard. A Slack notification is dispatched to your configured webhook with the full threat summary and enforcement outcome.

3. Threat Levels & Enforcement Behavior

Level Action Dashboard Color
HIGH Immediate block — VPC Firewall deny or Cloud Armor IP rule. Slack alert dispatched. red
MEDIUM Block applied. Slack alert dispatched. Incident flagged for review. amber
LOW / NONE No enforcement. Verdict cached for 24 hours to skip future analysis of the same IP. not shown

4. Slack Alert Configuration

Slack alerts are configured per organization during onboarding. Each alert includes:

  • Source IP and geographic location
  • Threat level and the Gemini model's explanation
  • Enforcement action taken (firewall rule name or Cloud Armor rule)
  • GCP project affected
  • Timestamp and incident ID for cross-referencing the dashboard

To update your Slack webhook URL, contact support or use the settings panel inside the Intelligence Portal.
