Getting Started_

Welcome to the Cybat Intelligence Engine. Activating your autonomous AI-driven security operations through the Google Cloud Marketplace takes under 10 minutes. Follow these steps to start protecting your GCP surface.

Prerequisites

Before subscribing, confirm your GCP environment meets the following requirements. Detection quality improves significantly with each item in place.

Required

  • ✓ Active GCP project with billing enabled
  • ✓ Cloud Logging enabled on your project
  • ✓ At least one internet-facing workload (GCE, GKE, or Cloud Run)
  • ✓ VPC network configured
  • ✓ Slack workspace + Incoming Webhook URL for alert delivery

Strongly Recommended

  • + Cloud HTTP(S) Load Balancer with access logging enabled — unlocks full L7 threat analysis
  • + Cloud Armor security policy attached to the Load Balancer — enables edge IP blocking
  • + Structured JSON logs with a source IP field — httpRequest.remoteIp or jsonPayload.ip

For the full list of supported log types and required fields, see the Security Specs page.

Step 1: Subscribe via Google Cloud Marketplace

  • Navigate to the Google Cloud Console and open the Marketplace.
  • Search for Cybat Intelligence Engine and open the product listing.
  • Select your billing account and the appropriate plan (Starter, Professional, or Enterprise), then click Subscribe.
  • Accept the Terms of Service. Once the subscription is confirmed, click Register with Cybat — you will be redirected to the Cybat Intelligence Portal.

Step 2: Sign In to the Intelligence Portal

Cybat uses Google Identity exclusively — no separate password is required.

  • At the portal login screen, click Sign in with Google.
  • Select your corporate Google Workspace account. Personal Gmail accounts are not authorized.
  • On first login, your Organization profile is automatically provisioned — a unique orgId is assigned to your account and linked to your GCP project billing identifier.
Open Intelligence Portal →

Step 3: Run the Provisioning Script

After logging in, the portal provides a single Google Cloud Shell command. Copy and run it in your GCP project. It takes approximately 60 seconds and performs the following actions automatically:

1.
Creates a Cloud Logging Sink — routes matching logs (HTTP, GKE, Cloud Armor) from your project to a dedicated Pub/Sub topic.
2.
Deploys a Pub/Sub Topic & Subscription — named soc-logs-{orgId}. The Cybat ingestion engine pulls from this subscription in real time.
3.
Grants minimal IAM roles to the Cybat service account: logging.viewer, compute.networkViewer, compute.securityAdmin (Cloud Armor enforcement), and compute.networkAdmin (VPC Firewall enforcement). No broader permissions are requested.

The engine applies a zero-retention policy — logs are analyzed in-stream and immediately discarded. No log data is stored on Cybat infrastructure.

Step 4: Verify Detection is Active

Return to the Intelligence Portal. Within 2–5 minutes of the provisioning script completing, you should see:

  • Your GCP project listed as Connected in the dashboard sidebar.
  • The 3D Threat Globe populating with activity from your infrastructure's geographic log sources.
  • Any detected threats appearing in the Incident feed with Gemini-generated analysis and enforcement status.

If no incidents appear within 10 minutes, confirm that Cloud Logging is enabled on your internet-facing workloads and that the Log Sink filter matches active log names.

Configuring Log Shipping

GKE and Cloud Run automatically forward logs to Cloud Logging. For all other workloads — VMs, bare-metal, and self-hosted Docker — you must configure log shipping explicitly. Choose the scenario that matches your setup.

Scenario A

GCE VM or On-Premise Server — Google Cloud Ops Agent

Required when your application runs directly on a Compute Engine VM or an on-prem server (connected via Cloud VPN). This covers: apps running natively on a VM, NGINX/Apache as a system service, or Docker containers on a VM that is not managed by GKE.

Install the Ops Agent

curl -sSO https://dl.google.com/cloudagents/add-google-cloud-ops-agent-repo.sh
sudo bash add-google-cloud-ops-agent-repo.sh --also-install

The VM's service account must have roles/logging.logWriter. Once installed, the Ops Agent automatically tails system logs and can be configured to tail your application's access log file via /etc/google-cloud-ops-agent/config.yaml.

Not needed for: GKE (Fluent Bit built-in), Cloud Run (stdout auto-shipped), App Engine (automatic).

Scenario B

Docker on a VM — gcplogs Log Driver

For Docker containers running on any VM or host (not GKE). Docker ships a built-in gcplogs driver that sends container stdout/stderr directly to Cloud Logging. The host VM's service account needs roles/logging.logWriter.

Option A — Per container (docker run)

docker run \
  --log-driver=gcplogs \
  --log-opt gcp-project=YOUR_PROJECT_ID \
  --log-opt gcp-log-cmd=true \
  your-image

Option B — docker-compose

services:
  app:
    image: your-image
    logging:
      driver: gcplogs
      options:
        gcp-project: YOUR_PROJECT_ID
        gcp-log-cmd: "true"

Option C — Default for all containers (/etc/docker/daemon.json)

{
  "log-driver": "gcplogs",
  "log-opts": {
    "gcp-project": "YOUR_PROJECT_ID"
  }
}

Then restart Docker: sudo systemctl restart docker

What Your Application Must Log

This requirement applies to every application you want Cybat to monitor, regardless of deployment platform. Cybat identifies threats from the source IP in each log record — without it, no analysis or enforcement is possible.

Log every inbound HTTP request received from external networks using Apache Combined Log Format or its structured JSON equivalent. Do not limit logging to errors only — the engine detects attack patterns across the full request history.

Apache Combined Log Format

$remote_addr - $remote_user [$time_local] "$request" $status $body_bytes_sent "$http_referer" "$http_user_agent"

# Example:
203.0.113.42 - - [30/Apr/2026:14:22:01 +0000] "GET /api/users HTTP/1.1" 200 1234 "-" "Mozilla/5.0"

Structured JSON (recommended — enables richer field extraction)

{
  "ip": "203.0.113.42",
  "method": "GET",
  "request": "/api/users",
  "status": 200,
  "bytes": 1234,
  "userAgent": "Mozilla/5.0",
  "timestamp": "2026-04-30T14:22:01Z"
}

Framework quick reference

  • NGINX — enable access_log with combined or JSON format
  • Node / Express — use morgan('combined') or a JSON middleware
  • Python / Flask or FastAPI — log request.remote_addr in a before-request hook
  • Go / Gin — use gin.Logger() or a custom access-log middleware

Do not log

  • Internal health-check endpoints (pod-to-pod traffic)
  • Traffic originating inside the VPC with no external IP
  • Plaintext unstructured lines with no parseable IP field
